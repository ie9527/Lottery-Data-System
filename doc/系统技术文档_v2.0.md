## 1. 数据更新流程与逻辑

### 1.1 数据源架构

系统依赖两个不同粒度的数据源：

| 数据源 | 配置文件位置 | 端点示例 | 数据范围 | 用途 |
|--------|------------|---------|---------|------|
| JSON API | `config.py#L30` | `https://www.17500.cn/inx/awards.html` | **仅最新一期** | 日常轮训增量更新 |
| TXT 文件 | `database.py#L103-112` | `http://data.17500.cn/3d_asc.txt` | **全量历史数据** | 断服补齐/首次导入 |

#### JSON API 结构（`config.py#L30`）

```python
JSON_API_URL = "https://www.17500.cn/inx/awards.html"
```

返回格式（按彩种ID索引的最新一期数据）：
```json
{
  "1": {"issue": "2025148", "numbers": "5,2,3", "date": "2026-06-05"},
  "2": {"issue": "2025148", "numbers": "3,1,4", "date": "2026-06-05"},
  ...
}
```

该端点**只包含每个彩种最新一期的数据**，不可用于历史数据回溯。

#### TXT 端点定义（`database.py#L103-112`）

```python
types_data = [
    ("3d", "3D", "福彩3D", "http://data.17500.cn/3d_asc.txt", ...),
    ("p3", "排列三", "排列三", "http://data.17500.cn/pl3_asc.txt", ...),
    ("p5", "排列五", "排列五", "http://data.17500.cn/pl5_asc.txt", ...),
    ("ssq", "双色球", "双色球", "http://data.17500.cn/ssq_asc.txt", ...),
    ("dlt", "大乐透", "大乐透", "http://data.17500.cn/dlt2_asc.txt", ...),
    ("qxc", "七星彩", "七星彩", "http://data.17500.cn/7xc_asc.txt", ...),
    ("7lc", "七乐彩", "七乐彩", "http://data.17500.cn/7lc_asc.txt", ...),
    ("kl8", "快乐八", "快乐8", "http://data.17500.cn/kl82_asc.txt", ...),

```

TXT 文件每行一条记录，以空格分隔字段。第一期到最新一期按升序排列。

### 1.2 端点合并逻辑

**核心机制**：JSON API 作为"增量检测器"，TXT 文件作为"全量数据源"，二者协同工作。

#### 合并流程（`auto_updater.py:catch_up_missing()`）

```
┌─────────────────────────────────────────────┐
│            catch_up_missing()                │
├─────────────────────────────────────────────┤
│                                             │
│  1. 调用 JSON API → 获取各彩种最新期号        │
│         │                                    │
│         ▼                                    │
│  2. 对比数据库中的最新记录                     │
│         │                                    │
│         ├─ 期号一致 → 跳过（数据完整）         │
│         │                                    │
│         └─ 期号不一致 → 检测到缺失             │
│               │                              │
│               ▼                              │
│  3. 下载对应彩种的 TXT 文件（全量）            │
│         │                                    │
│         ▼                                    │
│  4. 逐行解析 → INSERT OR IGNORE 补齐缺失      │
│         │                                    │
│         ▼                                    │
│  5. 提交事务                                 │
│                                             │
└─────────────────────────────────────────────┘
```

**关键函数**：
- `auto_updater.py#L137` — `catch_up_missing()`：断服补齐入口
- `auto_updater.py#L63` — `do_json_update()`：日常增量更新
- `fetcher.py#L21` — `fetch_json_api()`：调用 JSON API
- `fetcher.py#L59` — `fetch_data()`：下载 TXT 文件

#### 插入字段完整性

`catch_up_missing()` 从 TXT 文件插入数据时包含全部字段：

```sql
INSERT OR IGNORE INTO lottery_draws
(lottery_code, draw_number, draw_date, numbers, trial_numbers,
 machine_ball, draw_order, sale_amount, prize_pool, prizes, extra)
VALUES (?, ?, ?, ?, ?, ?, ?, ?, ?, ?, ?)
```

对比 `do_json_update()`（JSON API）插入的字段较少（JSON 数据本身不包含销售/奖池信息）：只插入 `numbers`、`trial_numbers`、`draw_order`，其余字段设为 NULL。

### 1.3 智能更新循环

`auto_updater.py:smart_update_loop()` 是后台自动更新守护协程。

#### 时间窗口配置（`config.py`）

| 参数 | 值 | 说明 |
|------|-----|------|
| `POLL_START_HOUR` | 21 | 开始轮训（开奖窗口起点） |
| `POLL_END_HOUR` | 23 | 安全终止时间 |
| `SAFETY_END_HOUR` | 23 | 强制结束 |
| `CHECK_INTERVAL` | 600 | 轮训间隔（秒）= 10 分钟 |

#### 循环逻辑

```
┌─────────────────────────────────────────┐
│          smart_update_loop()             │
│  (后台协程，服务器启动时自动启动)        │
├─────────────────────────────────────────┤
│                                         │
│  Loop:                                  │
│    │                                    │
│    ├─ 当前在活动窗口(21:00-23:00)？      │
│    │   ├─ No  → 休眠至 21:00             │
│    │   │       log: "非活动窗口"          │
│    │   │                                 │
│    │   └─ Yes → 进入活动窗口处理          │
│    │         │                            │
│    │         ▼                            │
│    │    查询今日开奖彩种列表               │
│    │         │                            │
│    │         ├─ 无开奖 → 跳过轮训          │
│    │         │   log: "跳过轮训"          │
│    │         │                             │
│    │         └─ 有开奖 → 进入内层循环      │
│    │               │                       │
│    │               ▼                       │
│    │          While True:                  │
│    │            │                          │
│    │            ├─ 已过23点？→ 终止        │
│    │            │                          │
│    │            ├─ 检查待更新彩种           │
│    │            │  (开奖时间+8分钟判断)     │
│    │            │                          │
│    │            ├─ 无待更新 → 轮训结束      │
│    │            │   log: "轮训结束"         │
│    │            │                          │
│    │            └─ 有待更新 → do_json_update│
│    │                  │                    │
│    │                  ├─ 有新增 → 记录      │
│    │                  │   log: "数据更新"   │
│    │                  │                     │
│    │                  └─ 无新增 → 等待重试   │
│    │                      log: "暂无新数据"  │
│    │                                         │
│    └───→ 休眠 CHECK_INTERVAL → 重新检查      │
│                                              │
└──────────────────────────────────────────────┘
```

#### 开奖时间判断（`draw_schedule.py:should_check_update()`）

```python
def should_check_update(lottery_code: str, last_updated_date: str | None) -> bool:
```

逻辑：
1. 获取该彩种今日的开奖时间配置
2. 如果 `last_updated_date` 等于今天 → 已更新，返回 False
3. 如果当前时间 > 开奖时间 + 8 分钟缓冲 → 可检查更新
4. 否则返回 False（等待开奖）

### 1.4 断服补齐机制

系统在以下时机触发数据补齐：

| 时机 | 调用位置 | 触发方式 |
|------|---------|---------|
| 服务器启动 | `main.py#L48` → `check_missing_data_on_startup()` | 自动 |
| 自动更新启动时 | `auto_updater.py#smart_update_loop()` 首轮 | 自动 |
| 手动触发 | `routes_api.py#POST /api/system/update/all` | 用户手动 |

`check_missing_data_on_startup()` 逻辑：

```
1. 获取所有活跃彩种的最新期号（从数据库）
2. 调用 JSON API 获取远端最新期号
3. 对比 → 如有缺失 → 调用 catch_up_missing() 补齐
4. 无缺失 → 跳过
```

### 1.5 数据解析流程

#### TXT 文件解析（`parser.py`）

每种彩种有对应的解析函数：
- `parse_line_3d()` → 3D（字段数 15）
- `parse_line_pl3()` → 排列三（字段数 15）
- `parse_line_pl5()` → 排列五（字段数 15）
- `parse_line_ssq()` → 双色球
- `parse_line_dlt()` → 大乐透
- `parse_line_qxc()` → 七星彩（字段数 23）
- `parse_line_qlc()` → 七乐彩
- `parse_line_kl8()` → 快乐八

**安全类型转换**（`parser.py:_safe_int()`）：

```python
def _safe_int(val, default=0):
    """安全转换为整数，无法转换时返回默认值"""
    try:
        v = int(val); return v
    except (ValueError, TypeError):
        return default
```

用于处理 TXT 数据中的 `-` 占位符（表示无此数据项）。

## 3. 工具调用提示词与规范

### 3.1 对话类型识别

系统提示词（`prompts.py:SYSTEM_PROMPT`）中有明确的类型识别规则：

```
## 对话类型识别（自动判断）
在回答问题前，先自动判断用户意图类型：
1. **闲聊/问候**（如"你好"、"谢谢"、"你是谁"）
   → 直接友好回复，不调用任何工具
2. **简单数据查询**（如"3D今天开什么"、"查询322出现次数"）
   → 调用 1~2 个工具即可
3. **复杂数据分析**（如"分析热号冷号"、"对比双色球和大乐透"）
   → 按需调用 2~4 个工具
4. **组合查询**（如"最近三次豹子号"、"3D和排列三都出现过什么豹子号"）
   → 使用合适的批量/组合工具
```

### 3.2 工具调用策略

**按需调用**原则：

| 场景 | 推荐工具 | 调用次数 |
|------|---------|---------|
| 查询最新开奖 | `get_latest_draws` | 1 |
| 查询某期开奖 | `get_draw_by_number` | 1 |
| 查询号码出现次数 | `get_number_stats` | 1~2 |
| 批量查询多个号码 | `batch_get_number_stats` | 1（替代逐个调用） |
| 查询号码历史明细 | `search_number` | 1~2 |
| 查询未开出号码 | `get_unused_numbers` | 1 |
| 日期范围查询 | `get_draws_by_date` | 1 |
| 时间段统计 | `get_draws_by_period` | 1 |
| 组选通配查询 | `get_group_selection_stats` | 1 |
| 和值/奇偶/跨度 | `get_complex_stats` | 1 |
| 热冷号分析 | `get_number_history` | 1~2 |
| 快乐八专项 | `get_kl8_data` | 1 |
| 跨彩种对比 | 各彩种分别调用 | 2~4 |
| 组合多条件 | 分步多工具 | 3~5 |

**全局上限**（`config.py:CHAT_TOOL_CALLS_LIMIT`）：**20 次**

### 3.3 工具调用终止条件

```
## 关键：何时停止调用工具（重要！）
1. **获取到足够数据后，立即合成回答**，不要反复调用工具
2. 如果工具返回的数据已能回答问题，直接生成最终回答
3. 最多调用 2~3 个工具就必须返回最终回答
4. 除非数据明显矛盾或缺失，否则不要继续调用新工具
5. 用户问的是具体问题，查一次 get_number_stats 或 search_number 就够
6. 每次工具调用后，检查返回结果，如果已经能回答用户问题，立即停止
```

### 3.4 豹子号专项策略

```
## 豹子号/特殊号码查询策略
1. 用户问"豹子号" → 调用 get_number_stats 查询
   000/111/222/333/444/555/666/777/888/999
   - 对于 3D/p3：一次 batch_get_number_stats 即可查完所有豹子号
2. 用户问"最近一次豹子号"：
   → 先 batch_get_number_stats 获取哪些豹子号存在
   → 然后 search_number 查明细 → 取最新日期
3. 用户问"某豹子号历史记录"：
   → 直接 search_number 查该号的开奖记录
4. **一次调用足够**：batch_get_number_stats 可以一次性查完 10 个豹子号
```

### 3.5 深度思考激活机制

| 条件 | 行为 |
|------|------|
| 用户勾选「深度思考」按钮 | 请求中 `thinking: true`，后端设置 `reasoning_effort` |

默认状态：**勾选（启用）**
未勾选时：`thinking: false`，后端不发送 `thinking` extra_body，不设置 `reasoning_effort`

### 3.6 缓存命中机制

```
1. 首次对话（某会话的第一条消息）：
   → prompt_cache_hit_tokens 强制设为 0（不显示缓存命中）
2. 后续对话（同一会话的第二条起）：
   → 正常使用 API 返回的 prompt_cache_hit_tokens 数据
```

**实现位置**：`router.py` 的 `chat_stream` 函数中维护 `_first_message_sessions` 集合。

### C. 数据库 ER 核心表

- `lottery_types` — 彩种定义（code, name, data_url, draw_count 等）
- `lottery_draws` — 开奖记录（lottery_code, draw_number, draw_date, numbers, sale_amount, prize_pool, prizes 等）
- `number_stats` — 号码统计预计算
- `number_dates` — 号码开出日期索引
- `update_log` — 更新日志
- `chat_sessions` — AI 对话会话
- `chat_history` — AI 对话历史
